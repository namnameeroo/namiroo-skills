# Create PR Workflow

이 스킬은 현재 변경사항을 프로젝트 표준에 맞춰 커밋하고 GitHub Pull Request를 생성합니다.

## 1. 정보 수집 및 분석

1. **사용자 확인**: 다음 정보를 확인합니다. (입력되지 않았다면 질문하세요)

- **티켓 번호**: (예: DPT-10557)
- **버전 태그**: `#major`, `#minor`, `#patch` (기본값: `#patch`)
- **Base 브랜치**: PR의 대상 브랜치
  - 자동 감지: `gh repo view --json defaultBranchRef --jq .defaultBranchRef.name`
  - `develop` 브랜치가 존재하면 `develop`, 없으면 default branch (`main`/`master`)를 기본값으로 사용
  - 사용자가 다른 브랜치를 지정한 경우 그것을 우선

2. **변경사항 분석**: 아래 명령어로 현재 작업 내용을 파악합니다.

- **미커밋 변경사항**: `git diff --staged` (현재 스테이징된 내용)
- **브랜치 간 차이**: `git diff {base_branch}...HEAD` (base 브랜치에서 갈라져 나온 후의 모든 코드 변경점)
- **커밋 히스토리**: `git log {base_branch}..HEAD --oneline` (이미 작성된 커밋 메시지들 확인)

> **Claude의 역할**: 위 3가지 정보를 종합하여 전체 작업의 맥락을 파악하고 PR 본문의 '작업 내용'을 상세히 작성합니다.

## 2. Commit 생성 (필요한 경우)

스테이징된 변경사항이 있다면 **Conventional Commits** 규격으로 커밋을 진행합니다.

- **커밋 메시지**: `{타입}: {요약} #{버전태그}\n\n- {주요 변경사항}\n\nCo-Authored-By: Claude {model_name} <noreply@anthropic.com>`
- **타입**: `feat` (기능), `fix` (버그), `refactor` (리팩토링), `docs` (문서), `chore` (기타)
- 이미 모든 작업이 커밋되어 있다면 이 단계를 건너뛰고 바로 Push로 진행합니다.

## 3. Push 및 PR 생성

1. **Push**: 현재 브랜치를 `origin`에 푸시합니다.

```bash
git push -u origin $(git rev-parse --abbrev-ref HEAD)

```

2. **PR 생성**: `gh` CLI를 사용하여 `{base_branch}` 브랜치를 대상으로 아래 템플릿에 따라 PR을 생성합니다.

```bash
gh pr create \
  --base {base_branch} \
  --draft \
  --assignee @me \
  --title "[{티켓번호}] {타입}: {요약}" \
  --body "{PR_BODY}"
```

### [PR Body 템플릿]

```markdown
# 🔗 티켓 링크

{티켓번호}

# 📋 작업 내용

- {변경사항 요약}

## 🧐 주요 검토 필요 사항

- {검토 포인트}

# 스크린샷

(생략)

# ✅ 체크리스트

{동적 체크리스트}
```

#### 체크리스트 동적 구성 규칙

PR 본문의 체크리스트는 **고정 문구가 아니라 변경사항 컨텍스트에 맞춰 매번 생성**합니다.
모든 항목은 `- [ ]` (빈 체크박스)로 둬서 작성자가 직접 확인 후 체크하도록 합니다.

**1) 기본 항목** (항상 포함)

- [ ] 코드 셀프 리뷰를 완료했다
- [ ] 변경 사이즈가 적절하다 (필요 시 PR 분할 고려)

**2) 변경사항 유형별 조건부 항목**

변경된 파일/내용을 분석해 해당되는 항목만 추가합니다.

| 변경 유형 감지 단서 | 추가할 체크리스트 항목 |
|---|---|
| `*.test.*`, `*.spec.*`, `__tests__/` 수정 | `- [ ] 테스트가 로컬에서 통과한다` |
| 비테스트 로직 변경 (테스트 미동반) | `- [ ] 신규/변경 로직에 대한 테스트를 추가했거나, 미추가 사유를 본문에 적었다` |
| UI/스타일 변경 (`*.tsx`, `*.scss`, `*.css`, 컴포넌트) | `- [ ] 변경 전/후 스크린샷 또는 영상을 첨부했다` |
| API 시그니처/스키마 변경 (`*.controller.*`, `*.dto.*`, OpenAPI/Swagger) | `- [ ] API 호환성을 확인했고, 클라이언트 영향도를 PR 본문에 기술했다` |
| DB 마이그레이션 (`migrations/`, `*.sql`, schema 파일) | `- [ ] 마이그레이션 롤백 절차를 검토했다` |
| 환경변수 추가/변경 (`.env.example`, config 파일) | `- [ ] 신규 환경변수를 README/문서에 기록했고 인프라 팀에 공유했다` |
| 의존성 변경 (`package.json`, `pnpm-lock`, `requirements*.txt`) | `- [ ] 신규 의존성의 라이선스/보안 이슈를 확인했다` |
| 성능 민감 코드 (대량 데이터 처리, 쿼리, 렌더링 핫패스) | `- [ ] 성능 영향(전/후 수치)을 측정했거나, 영향 없음을 본문에 명시했다` |
| 보안 민감 코드 (auth, 권한, 토큰, 암호화) | `- [ ] 보안 영향 범위를 본문에 명시했고 셀프 리뷰에서 점검했다` |
| 문서만 수정 (`*.md`, `docs/`) | `- [ ] 링크/예제 코드가 실제 동작한다` |
| breaking change (메이저 버전 bump, public API 제거/변경) | `- [ ] 마이그레이션 가이드를 본문 또는 별도 문서에 작성했다` |

**3) 구성 절차**

1. `git diff {base_branch}...HEAD --name-only`로 변경 파일 목록을 얻는다
2. 위 표의 단서와 매칭해 해당하는 항목만 선택
3. 기본 항목(1) + 매칭된 조건부 항목(2)을 합쳐 PR 본문의 체크리스트 섹션을 작성
4. 매칭 결과가 모호하면 사용자에게 "이 항목 포함할까요?" 짧게 확인 (3개 이내)

**예시** — React 컴포넌트 + 테스트 추가 PR이면:

```markdown
# ✅ 체크리스트

- [ ] 코드 셀프 리뷰를 완료했다
- [ ] 변경 사이즈가 적절하다 (필요 시 PR 분할 고려)
- [ ] 테스트가 로컬에서 통과한다
- [ ] 변경 전/후 스크린샷 또는 영상을 첨부했다
```

## 4. PR 생성 후 검증

PR 생성 후 반드시 CLI로 실제 생성 여부를 검증합니다:

### 검증

1. 방금 만든 PR이 존재해야 함

```bash
gh pr view --json url
```

2. 생성한 URL의 organization이 실제 CLI 실행결과와 일치해야함

```
git remote get-url origin
```

### 검증 성공 시

- PR URL과 번호 출력
- 다음 단계 안내 (리뷰 요청, 리뷰 코멘트 검토, worktree 정리 등)

### 검증 실패 시 (PR이 생성되지 않은 경우)

사용자에게 직접 실행할 수 있는 명령어를 제공합니다:

```
gh pr create --base {base_branch} --title "[{티켓번호}] {타입}: {제목}" --body "{PR 본문}"
```

**중요** : PR 생성 결과를 추측하지 말 것. 반드시 gh pr view 명령어로 확인한 실제
결과만 사용자에게 전달할 것.

## 5. Worktree 및 특이사항 지원

- **Worktree**: `git worktree list`를 확인하여 워크트리 환경인 경우 해당 경로의 컨텍스트를 유지합니다.
- **이미 커밋된 경우**: 스테이징된 변경사항이 없다면 최근 커밋 메시지를 분석하여 PR 생성 단계로 바로 넘어갑니다.
